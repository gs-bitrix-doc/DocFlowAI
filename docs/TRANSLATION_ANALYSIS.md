# Анализ качества перевода

## Текущее состояние *(обновлено 07.04.2026)*

Инструмент работает стабильно. Типичный API-документ переводится чисто — без кириллицы в выводе, структура сохраняется полностью. На файлах разного размера и сложности система показывает предсказуемый результат.

**Что работает хорошо:**
- Структура документов сохраняется: ссылки, якоря, таблицы, блоки кода, вкладки
- Кириллица в примерах кода (строки в кавычках, инлайн-код) переводится отдельным шагом
- Ячейки YFM-таблиц с типами (`[`string`](url) | описание`) восстанавливаются автоматически если LLM их сломал
- Если после основного перевода в тексте осталась кириллица — она дополнительно переводится отдельным запросом
- При сетевых проблемах или временной недоступности API — автоматические повторные попытки
- Типичные файлы (~4000–9000 симв.) проходят валидацию без ошибок и предупреждений

**Единственная рабочая модель:** `bitrixgpt-5.5`. Модель `gpt-oss-120b` несовместима — разворачивает плейсхолдеры независимо от инструкций, ломает структуру таблиц.

**Известные ограничения:**
- Очень большие файлы (18 000+ симв., 140+ плейсхолдеров) иногда дают нестабильные ячейки в конкретных полях — это особенность модели на длинных документах, не системная проблема
- Валидатор может давать ложные предупреждения по количеству строк таблиц если оригинал использует нестандартное форматирование `||` (на отдельной строке) — содержимое при этом правильное

## 07.04.2026 — document-generator-region-add, серия прогонов

**Файл:** `document-generator-region-add.md`, 8886 симв. Особенность: одно и то же поле (`formatName`) присутствует в двух разных таблицах — как многострочная ячейка в таблице параметров и как однострочная в таблице ответа.

**Что исправили в ходе прогонов:**

В `fix_cell_types` (восстановление ячеек типов) был баг: карта соответствий хранила одну запись на поле, из-за чего при одинаковых именах полей в разных таблицах логика применяла неправильный оригинал. Исправлено: теперь хранится список всех вхождений, N-е в переводе сопоставляется с N-м в оригинале. Также добавлен обратный случай: если LLM удалил `||` у однострочной ячейки — он восстанавливается.

В `TranslationFixer` (доперевод кириллицы) не обрабатывались начала многострочных ячеек вида `` [`тип`](url) | описание `` (без `||`). Исправлено: такие строки теперь тоже переводятся — LLM получает полную строку для контекста, из ответа извлекается только описание.

**Финальный результат (test_8):** 0 ошибок, 0 предупреждений. Оба `formatName` — в таблице параметров и в таблице ответа — корректны.

## 03–04.04.2026 — фундамент

Заложена основная архитектура пайплайна: статические замены (PreTranslator), защита структуры плейсхолдерами (Protector), перевод строк в коде (CodeTranslator), валидация после перевода. Определена единственная рабочая модель — `bitrixgpt-5.5`.
