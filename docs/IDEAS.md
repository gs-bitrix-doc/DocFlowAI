# Ideas & Roadmap

## Пайплайн: что ещё не реализовано

Текущий пайплайн умеет переводить, чинить структуру и делать retry. Три модуля, которых пока нет:

| Модуль | Что делает |
|---|---|
| `StyleValidator` | LLM проверяет стиль и структуру документа до перевода — нет смысла переводить плохо написанный оригинал |
| `DictionaryFixer` | Находит непереведённые термины из словаря и заменяет напрямую, без LLM — в отличие от `TranslationFixer`, работает с контентом, а не со структурой |
| `LLMFixer` | Если после `DictionaryFixer` остался русский текст — отдаём его в LLM на доперевод |

## DocFlow Web

Сейчас пайплайн — инструмент для разработчика. DocFlow Web превращает его в продукт для команды.

**Идея**: веб-платформа, где перевод запускается автоматически при каждом пуше в русский репозиторий, а команда работает через браузер — смотрит логи в реальном времени, правит результат и нажимает «Опубликовать».

### Как выглядит рабочий процесс

1. Разработчик пушит изменения в RU репозиторий
2. GitHub webhook триггерит перевод — автоматически, без ручного запуска
3. Техпис открывает браузер, видит список задач и их статусы
4. Переводчик смотрит diff оригинал/перевод, при необходимости вносит правки
5. Клик «Опубликовать» — изменения уходят прямым коммитом в EN репозиторий

Между шагом 1 и 5 — ноль ручных действий от техписа.

### Архитектура

```
GitHub (RU репо)
    ↓ webhook при каждом пуше
Node.js + Express
    ↓ создаёт Job, ставит в очередь
Python пайплайн (child_process)    ←→    MongoDB (Jobs, логи, правки)
    ↓ логи стримятся через SSE
Браузер
    ↓ редактирование + «Опубликовать»
GitHub API → прямой коммит в EN репо
```

### Стек

- **Backend:** Node.js + Express
- **БД:** MongoDB + Mongoose
- **GitHub:** Octokit — webhook на входе, прямой коммит на выходе
- **Стриминг логов:** SSE (Server-Sent Events) — результат виден в реальном времени
- **Frontend:** React

### Модель данных

```
// Job — одна задача перевода
{
  sourceRepo, targetRepo, filePath, commitSha,
  sourceContent,        // оригинал
  translatedContent,    // результат пайплайна
  editedContent,        // правки редактора (если были)
  status: 'pending' | 'translating' | 'review' | 'published' | 'failed',
  logs: [{ level, message, timestamp }],
  publishedCommitSha,
  publishedAt,
  publishedBy,          // ref: User
}
```

### Интересные технические задачи

- **Стриминг логов** — SSE позволяет видеть прогресс перевода в реальном времени без polling
- **Очередь** — `p-limit` на 2-3 параллельных job'а, чтобы не перегружать LLM API при массовых пушах
- **Конфликты** — если EN файл правился вручную пока шёл перевод, нужно это обнаружить и показать diff

### Итерации

1. **MVP:** webhook → перевод → просмотр логов в браузере
2. **Редактор:** diff оригинал/перевод, inline правки
3. **Публикация:** кнопка «Опубликовать» → коммит в EN репо
4. **Команда:** роли admin/editor, история публикаций

## Другие идеи

### Кэш переводов (Translation Memory)
Сохранять уже переведённые фразы и абзацы. Если тот же текст встречается
в другом файле — брать из кэша без запроса к API. Экономия токенов и единообразие терминологии.

### Инкрементальный перевод
Переводить только то, что изменилось с прошлого раза через `git diff`.
