# DocFlow AI

Инструмент для автоматического перевода `.md` файлов документации с сохранением структуры.

## Как запустить

**1. Установить зависимости:**
```bash
uv add requests python-dotenv click mistune
```

**2. Заполнить `.env`** (скопировать из `.env.example`):
```env
API_KEY=your_api_key_here
BASE_URL=https://your-bitrix-gpt-endpoint
MODEL=your_model_name
```

**3. Запустить:**

```bash
# Перевести один файл
python main.py input/имя файла

# Перевести всю папку
python main.py --dir input/

# Проверить пайплайн без перевода (без API ключа)
python main.py input/crm-deal-get.md --dry-run

# Тестовый режим — файлы нумеруются: test_1_..., test_2_...
python main.py input/crm-deal-get.md --test
```

Результат сохраняется в `output/`:
```
input/crm-deal-get.md → output/crm-deal-get.en.md
input/crm-deal-get.md → output/test_1_crm-deal-get.en.md  # с --test
```

### Отладка пайплайна

```bash
# Посмотреть что попадает в LLM (шаги 1-4 без API)
python tools/inspect_pipeline.py input/crm-deal-get.md --no-code-translate --output debug_output.txt
```

## Пайплайн

```
Файл → Parser → PreTranslator → CodeTranslator → Protector → Translator
     → Protector.restore → fix_cell_types → TranslationFixer → Validator → Writer
```

## Модули

### `src/processors/parser.py` — MarkdownParser

Читает `.md` файл, проверяет что файл существует и имеет расширение `.md`.

---

### `src/processors/pre_translator.py` — PreTranslator

Выполняет статичные замены **до** того, как Protector скроет текст от LLM.
Работает без обращения к API. Все словари загружаются из `data/pre_translator/*.json`.

**Принцип:** статикой покрываем только то, что LLM переводит нестабильно или где важна точность. Всё остальное переводит LLM.

**Что заменяет:**

| Что | Пример | Почему статично |
|---|---|---|
| Метки include-блоков | `[Сноска о параметрах]` → `[Note on parameters]` | Определяется по имени файла |
| Заголовки нот и cut-блоков | `"Развитие метода остановлено"` → `"Method development has been halted"` | Стандартные шаблоны; LLM не видит их без статической замены |
| Заголовки разделов | `## Продолжите изучение` → `## Continue learning` | LLM переводит нестабильно |
| Заголовки типов элементов | `### Тип элемента data` → `### data item type` | Стандартный паттерн |
| Заголовки типов | `### Тип params` → `### params type` | Стандартный паттерн; только если после слова латинский идентификатор |
| Заголовки параметров | `### Параметр params` → `### Parameter params` | Стандартный паттерн |
| Заголовок YFM-таблицы | `Название / тип` → `Name / type` | LLM схлопывал строки |
| Названия колонок таблиц | `**Код** \| **Описание** \| **Значение**` → `**Code** \| **Description** \| **Value**` | LLM даёт разные варианты |
| `HTTP-статус:` | → `HTTP status:` | LLM варьирует регистр |
| `` `тип` `` | → `` `type` `` | Всегда одно и то же |
| `![alt-текст]` | → `![alt text]` | Placeholder без смыслового содержания |
| Термины из `static_terms.json` | `элемент` → `item`, `смарт-процесс` → `SPA` | LLM игнорирует словарь для этих слов |

**Заголовки нот и cut-блоков:** если заголовок есть в `note_titles.json`, заменяется статично. Если нет — остаётся на месте и переводится LLM (Protector не трогает содержимое заголовка, только синтаксис тега).

---

### `src/processors/code_translator.py` — CodeTranslator

Отдельный LLM-вызов **до** Protector. Находит кириллические строковые значения внутри кодблоков и инлайн-кода, переводит их двумя отдельными батчами:

- **Батч 1** — строки в кавычках внутри блоков кода (двойные и одинарные)
- **Батч 2** — инлайн-код вне кодблоков (`` `текст` ``)

Разделение батчей предотвращает смешение контекстов. Кодблоки после этого защищаются Protector целиком — структура не страдает.

---

### `src/processors/protector.py` — MarkdownProtector

Защищает элементы которые нельзя переводить — заменяет их на плейсхолдеры `[[PROTECTED_N]]` перед переводом и восстанавливает после.

**Порядок защиты (важен):**

| Приоритет | Что защищаем | Пример |
|---|---|---|
| 1 | Блоки кода | ` ```js\ncode\n``` ` |
| 2 | Теги note/cut с заголовком | `{% note tip "Title" %}` → синтаксис защищён, заголовок виден LLM |
| 3 | Frontmatter | `---\ntitle: ...\n---` |
| 4 | Битрикс-блоки без заголовка | `{% list %}`, `{% if %}`, `{% endif %}` |
| 5 | Названия полей API | `**TYPE_ID**`, `**CATEGORY_ID**` |
| 6 | Якоря | `{#deal}`, `{#anchor-name}` |
| 7 | Картинки | `![alt](url)` |
| 8 | Лейблы вкладок | `- cURL (OAuth)`, `- JS` внутри `{% list tabs %}` |
| 9 | Ссылки (URL) | `[текст](url)` — защищается только URL, текст остаётся |
| 10 | Инлайн-код | `` `crm.deal.get` `` |

**Принцип:** синтаксис защищён, содержимое видимо для LLM. Применяется к ссылкам (URL скрыт, текст открыт) и к note/cut тегам (синтаксис скрыт, заголовок открыт).

**Инварианты:**
- Теги note/cut обрабатываются до общего `{%...%}` паттерна, иначе общий паттерн поглотит их целиком.
- Ссылки защищаются раньше инлайн-кода. Иначе `` [`type`](url) `` → плейсхолдер содержит `]` → ломает link-паттерн.

---

### `src/processors/translator.py` — Translator

Отправляет текст в Bitrix GPT (LiteLLM-совместимый API) и возвращает перевод.
После получения ответа заменяет NBSP (`\u00a0`) на обычные пробелы.

**Retry-логика:** при 5xx-ошибках и сетевых проблемах делает до 3 повторных попыток с задержками 5 / 15 / 30 секунд. Ошибки 4xx поднимаются сразу (неверный запрос, не решается повтором). После исчерпания попыток бросает `RuntimeError`.

**Модели:**
- `bitrixgpt-5.5` — основная модель

---

### `src/processors/writer.py` — MarkdownWriter

Сохраняет результат в папку `output/`. В тестовом режиме (`--test`) нумерует файлы автоматически: `test_1_file.en.md`, `test_2_file.en.md` и т.д.

---

### `src/validators/structure.py` — StructureValidator

Сравнивает структуру оригинала и перевода после обработки.

**Что проверяет:**

| Проверка | Тип |
|---|---|
| Плейсхолдеры восстановлены | error |
| Количество заголовков `##` | error |
| Количество таблиц `#\|` / `\|#` | error |
| Строки таблиц `\|\|` | error |
| Количество `\|\|` в таблицах (cell closers) | warning |
| Блоки кода ` ``` ` | error |
| Битрикс-блоки `{% %}` | error |
| Якоря `{#...}` сохранены | error |
| Внутренние ссылки `(#...)` не битые | warning |
| Количество ссылок `[text](` не уменьшилось | warning |
| Термины из словаря переведены | warning |
| Сломанная ячейка типа `` [`type` `` без `](url)` | warning |
| Кириллица вне кодблоков (с фрагментом) | warning |
| Запрещённые домены, бренды, телефоны | warning |

Термины словаря проверяются по точным границам слова (`\b`), не по подстроке.
При ошибках файл всё равно сохраняется, в лог пишется предупреждение.

**Важно:** `_strip_code` удаляет кодблоки и инлайн-код перед проверкой содержимого. Паттерн инлайн-кода использует `[^\`\n]` (без переноса строк) чтобы не съедать `||`-разделители через несколько строк.

---

### `src/fixers/cell_type.py` — fix_cell_types

Восстанавливает строки типа в YFM-ячейках по оригиналу. Запускается после `Protector.restore`, до `TranslationFixer`.

**Алгоритм:**
1. Из оригинала строит карту `field_name → [type_line, ...]` — список всех вхождений в порядке появления
2. В переводе обходит те же поля, отслеживая номер вхождения (N-е в переводе → N-е в оригинале)
3. Если строка типа сломана (LLM удалил ссылку) — восстанавливает из оригинала
4. Если оригинал не закрывал ячейку (`||` нет), а перевод добавил — удаляет лишний `||`
5. Если оригинал закрывал ячейку (`||` есть), а перевод удалил — добавляет обратно

Позиционное сопоставление решает проблему одноимённых полей в разных таблицах (например, `formatName` как многострочная ячейка в таблице параметров и как однострочная в таблице ответа).

---

### `src/fixers/translation.py` — TranslationFixer

Доперевод оставшейся кириллицы после основного LLM-перевода. Запускается после `fix_cell_types`, до валидатора. В `--dry-run` режиме не вызывается.

**Алгоритм:**
1. Сканирует перевод построчно (вне кодблоков), собирает строки с кириллицей
2. Для однострочных YFM-ячеек (`` [`тип`](url) | описание || ``) — отправляет полную строку в LLM для контекста, из ответа извлекает только описание (структура не ломается)
3. Для начала многострочных ячеек (`` [`тип`](url) | описание `` без `||`) — аналогично, но без `||` в суффиксе
4. Для обычных строк — переводит целиком
5. Если после перевода строка всё ещё содержит кириллицу — пропускает (не ухудшает)
6. Применяет замены через `re.sub` с точным совпадением строки

Батч-запрос к LLM: все уникальные строки за один вызов, кастомный короткий промпт.

## Настройки перевода

- Направление: **RU → EN**
- Промпт: `data/prompt.txt` — редактируется без изменения кода
- Основной словарь: `data/dictionary.json` — попадает в промпт LLM и используется валидатором
- Расширенный словарь: `data/glossary.json` — 903 термина, только для валидатора
- Статичные замены: `data/pre_translator/*.json` — редактируются без изменения кода
